Diagnosis before build. Stage 1 started from an existing scraper, not from scratch. The job board integration, deduplication, and seen-jobs tracker were already functional. The constraint was signal, not infrastructure. Two hours of evaluation determined what to keep and what to discard before a line of original code was written.
Prompt calibration over feature expansion. Stage 2 produced a scoring layer that returned MAYBE on nearly everything. The fix was one instruction added to the prompt: "Do not hedge. Make a call." Output quality improved immediately. The lesson applied to every subsequent LLM interaction: the prompt is the product. Infrastructure is commodity.
Hard gates over weighted scoring. Stage 4 replaced a three-criterion rubric with an explicit QUALIFY/NEUTRAL/DISQUALIFY framework. The disqualify list (requires Apex, French required, global SI employer, US-only authorization) produced sharper results than any weighting scheme. Concrete rules outperform flexible scoring when the constraints are well-understood.
Funnel visibility as a feature. Stage 3 added a metrics line to the email digest: scraped → filtered → scored → relevant. This was a late addition that became one of the most useful outputs. When something breaks, rate limiting, over-filtering, or misconfiguration, the funnel surfaces it immediately without reading logs.
Designing for Outlook, not against it. Stage 7 spent multiple commits fighting Outlook's rendering engine before the correct diagnosis: Outlook ignores CSS background-color on divs entirely and applies its own dark-mode transformations on top of explicit bgcolor attributes. The fix was to stop treating Outlook as a bug and design the default palette for it. Light mode as default, dark mode as enhancement. The same diagnostic pattern applied here as in a broken CRM: understand the constraint before proposing the fix.
README written for the person who has never opened a terminal. Stage 6 documented six real errors from the first external user run. None were hypothetical. The PowerShell execution policy block, the missing Workflows permission on the GitHub token, the exact string required for location values. Every step that can fail was documented because it had already failed for someone.
Test infrastructure before production. Stage 8 built two modes before the tool went to a second user: --preview sends a sample email with no API calls, --test runs the full pipeline at reduced scale. The Outlook rendering problem was caught in test mode because test mode sends the actual HTML email. Terminal-only output would have hidden it until the first real run.
Confidence as a separate output from decision. Stage 12 onward treats YES/MAYBE/NO and HIGH/MEDIUM/LOW as independent signals. A MAYBE with HIGH confidence is a real borderline fit. A MAYBE with LOW confidence is a posting without enough information to decide. Collapsing those into a single score loses the distinction a hiring manager or a well-built lead scoring model would preserve.
Hypothesis split into two signals. Stage 19 separated a single blended hypothesis sentence into two labelled outputs: why the company is hiring, and what specific value the candidate brings to address that problem. A Backfill hire requires a continuity argument. A Recovery hire requires a turnaround record. A New capability hire requires evidence of building without a playbook. A single sentence cannot carry all three without becoming generic.
Forced-choice categorization over free-form inference. The six hypothesis categories (Backfill, Capacity, New capability, Recovery, Strategic bet, Unclear) were defined before the prompt was written. Free-form inference produces inconsistent output. A forced choice produces consistent, actionable signal. The same principle applies to Salesforce picklist design: constrain the input, improve the output.
Ghost detection weighted asymmetrically. Career page found is strong positive signal. Career page not found is weak signal. Companies post on LinkedIn without updating their careers page regularly. The asymmetry is intentional. Repost history is the most reliable ghost signal because it comes from observed behavior over time, not from a single HTTP check. A tool that claims high confidence from weak signals is worse than one that admits uncertainty.
Dead code deleted, not commented out. Stage 18 removed the Telegram bot, resume generator, Flask dashboard, and network tracker entirely. Each had been built and abandoned. Commenting them out would have left maintenance surface and confused every future reader. Git history preserves the record. The active codebase should tell a coherent story.
Config-driven, not code-driven. The qualification logic, candidate profile, search parameters, and scoring rules live in one YAML file. The code reads the file; it does not contain the logic. This is the same principle that makes a well-built Salesforce org maintainable: business rules should be configurable by someone who understands the business, not hardcoded by someone who understands Python.
Refactoring sequenced deliberately. Stage 18 split a 920-line monolith into four modules. The order mattered: config naming fixed before the split, logging unified before the split, dead code deleted before the split. Each dependency resolved before the change that depended on it. The wrong order would have locked in the naming inconsistencies or carried dead imports into the new files.
Single source over multiple unreliable ones. LinkedIn was the only source that consistently produced parseable results. Indeed and Glassdoor scraped inconsistently. Two hours of comparative testing produced a decision that a feature flag would have obscured: only ship the source that works. The config comments explain why the others exist but are disabled.
Seen-jobs persistence via Actions cache. GitHub Actions runs in an ephemeral container that discards all state on completion. The seen-jobs SQLite database is uploaded as an artifact after each run and restored at the start of the next. Without this, every run would re-score jobs already seen and re-send postings already emailed.
Job postings as ideals, not minimums. A scoring architecture that treats every stated requirement as a hard gate produces too many NO results on postings written by HR generalists copying from the last hire cycle. Years of experience, preferred certifications, and seniority titles are signals the scorer weighs against demonstrated outcomes. Only the disqualify list creates hard gates.
Notification in every scenario. A run that finds nothing is not a silent run. A no-results email is sent with the funnel metrics so the operator knows the pipeline ran and nothing passed, rather than wondering whether it ran at all. Two distinct code paths existed: no new listings (caught early) and no relevant scores (was silent). Both now send.
Separate public surface from personal config. The public repo contains an example config with generic placeholders. Personal data lives in config.yaml, gitignored via .git/info/exclude. In GitHub Actions, config.yaml is reconstructed from a secret at runtime. This allows the project to be public and functional without exposing personal data, salary floors, or location constraints.
Evidence extraction as anti-hallucination constraint. The scoring prompt requires the LLM to include verbatim quotes from the job posting that support its decision. If no evidence is found, confidence is downgraded. The constraint catches hallucinated requirements the posting never stated, and makes every score auditable without re-reading the posting.
Tiered signal architecture. A flat rubric produces inconsistent results because not all signals carry equal weight. A tiered structure with core, strong, and supporting qualify signals, categorized disqualify signals, and a separate neutral band gives the LLM a model that matches how a hiring manager actually reads a posting. The decision framework rules are explicit: any disqualifier triggers NO, three or more qualify signals trigger YES, partial signals produce MAYBE.
Hypothesis category load-bearing for both outputs. The six forced-choice categories (Backfill, Capacity, New capability, Recovery, Strategic bet, Unclear) carry the causal frame that makes both hypothesis halves accurate. A Backfill hire requires a continuity argument. A New capability hire requires evidence of building without a playbook. The category is not a label. It is the logical premise for the value statement that follows.
Observed behavior proves hypothesis better than posting language. Company tracking data feeds into the hypothesis before the LLM makes a category decision. A single posting that uses growth language is ambiguous. Three identical titles posted by the same company in the same month is a Capacity signal, regardless of what the posting says. Repost count, active role count, and title similarity are behavioral signals, not marketing copy.
DNS resolution not covered by request timeouts. requests.timeout=5 sets a socket timeout but does not cap the DNS resolution phase on Windows. A careers page probe on a nonexistent domain could stall for 60 seconds or more before failing. The fix was a ThreadPoolExecutor wall-clock timeout wrapping the entire probe function, not the socket timeout.
Parallel probing covers domain variants. A company named "Ridge Partners" may operate from ridgepartners.com, ridge-partners.com, ridgepartners.io, or ridge-partners.ca. Probing sequentially with timeouts would take too long to be useful. Generating all variants and firing them in parallel, returning on the first 200 response, cuts wall time while increasing hit rate.
Name as positioning decision. The original name was "Automated Job Search Digest." It described the mechanism. The final name, Pathfinder: Cut the Job Search BS, describes the outcome and the stance. Documentation, commit messages, and email subjects updated accordingly. A tool people use daily should be named like something they want to use.
Stepping stone treated as a valid path. Roles that are not exact-fit consulting roles but build directly toward consulting (CRM administration, RevOps, solutions architecture, technical pre-sales) are scored as supporting qualify signals rather than neutral or disqualifying. The scoring logic reflects a job search strategy, not a job description match. Trajectory matters.
